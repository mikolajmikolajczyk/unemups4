## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
After the white-dummy fix (merge efa37b0), Celeste's previously-black content renders its VERTEX COLORS (geometry visible), but NOT the real texture atlas — the game art is still missing. Root: the content sprites (incl. the count=300 title/mountain batch) sample an InlineVSharp{sgpr=0} T# that decodes to a degenerate 2x1 placeholder whose base is UNMAPPABLE (garbage in the guest's SGPR block; established task-153 + RenderDoc frame6 EID45). Two possibilities: (1) these are genuinely FNA/XNA SpriteBatch solid-color / vertex-tinted draws that use a 1x1 white dummy -> white-dummy is correct and vertex-color IS the intended output (then the missing 'art' is elsewhere / a later scene); (2) the guest supplies the real atlas T# via a descriptor mechanism we don't follow (SetPointer / a descriptor table indexed off a different SGPR), and we mis-read it as InlineVSharp -> garbage -> we whiten it, masking the real atlas. Investigate: for a content draw known to need a real texture (e.g. the mountain_wall or a sprite atlas), dump the guest's actual descriptor setup (which SGPRs hold the T#/the descriptor-set pointer, whether it's inline or a memory deref), compare vs how derive_texture_binding classifies DescriptorSource (InlineVSharp vs SetPointer, crates/gnm/src/exec.rs / vbuf.rs derive_texture_inline). Relates efa37b0 (white-dummy), task-153 (the 2x1 T#), doc-6. If (1), close as 'correct, art is a later scene'; if (2), fix the descriptor provenance so the real atlas reaches these draws.

## Phase 5 (2026-07-18, worktree agent-a24b7b60b8c471cd4) — ROOT CAUSE CONFIRMED + FIXED

Ran the falsifiable GPU-completion-latency experiment. **VERDICT: CONFIRMED — the collapse is caused by our
SYNCHRONOUS EOP memory-fence completion, and it is FIXED (Celeste renders the fully textured splash by default).**
This SUPERSEDES the earlier task premise (provenance/InlineVSharp) AND the Phase-1..4 "managed-heap `_dirty`"
narrowing: the divergence is NOT the managed `_dirty` gate but the NATIVE gnmx command-buffer-recycle path, which
polls the EOP memory fence.

**(1) GPU-completion signaling map.** The guest's per-frame GPU sync: `sceGnmAddEqEvent(type=64 EOP, id=0)` once +
`sceKernelWaitEqueue` every frame (drains `equeue::signal_gpu_completion`, fired on `sceGnmSubmitDone`) — this is
the guest's frame-sync primitive. SEPARATELY, every flip DCB carries one `IT_EVENT_WRITE_EOP` that writes a
**memory fence** (value 0) to one of TWO per-context addresses ping-ponging by frame parity
(`0x902046a08` even / `0x902c46a10` odd — the double-buffered contexts). Our executor writes this fence INLINE in
`exec.run` (`write_eop_label`→`write_label`), before the submit HLE returns → instant per-buffer completion.

**(2) Experiment sweep (per-frame 8-dword T#@0x2c0c on the flip DCB; baseline = 3,3,0,0,0…):**
- Defer the EQUEUE signal (`UNEMUPS4_GPU_LATENCY` wall-clock, 1/2/6/30): **NO CHANGE** (3,3,0,0,0 even when 9/30
  waits forced to observe "GPU not done"). Equeue is NOT the lever.
- Defer the EOP MEMORY FENCE (wall-clock or submit-count `UNEMUPS4_GPU_LATENCY_SUBMITS=D`): collapse shifts out
  by exactly the latency — `collapse frame = D + 2` (D=1→@3, D=2→@4, D=8→@10). Fence IS the lever.
- WITHHOLD the fence (never mark done): **3,3,3,… EVERY frame, matching real HW, no deadlock.**

**(3) VERDICT: CONFIRMED = GPU-completion latency, specifically the EOP MEMORY FENCE (not the equeue).** gnmx
recycles its double-buffered command contexts by CPU-polling this fence; instant completion → fast recycle path
that skips re-emitting the per-draw state (incl. the atlas T#/S#). No fixed finite latency sustains it at steady
60 Hz (the reused buffer's fence always reads "done" by the recycle check).

**(4) FIX (`crates/gnm/src/exec.rs`).** Split emit from store: new `emit_label` DEFAULTS to pipelined completion —
does NOT surface the EOP/EOS memory fence synchronously (the guest's real sync is the equeue, still signaled).
`UNEMUPS4_GPU_EOP_SYNC=1` restores the pre-fix inline write (A/B + any title that CPU-polls the memory label as
its only completion signal). SELF-VERIFY: default (no env) → DCB `3,3,3,…` + PNG shows the TEXTURED
"Matt Makes Games Inc. / presents" splash (gold→purple gradient, outlined font atlas, animated bokeh — READ the
PNG, live/animating not frozen); `UNEMUPS4_GPU_EOP_SYNC=1` → DCB `3,3,0,0,0` + white bar. Build/clippy/test green
(ps4-gnm 207 tests; workspace green). doc-6 Entry 20 + doc-5 Case 24 + doc-4 taxonomy row added.

**Residual / caveat for maintainer review.** The fix changes the DEFAULT EOP-fence behavior for ALL titles. It is
safe for Celeste (which syncs via the equeue), but a homebrew that CPU-polls the raw EOP memory label as its ONLY
GPU-completion signal (no equeue) would no longer observe completion → could stall; such a title must set
`UNEMUPS4_GPU_EOP_SYNC=1`. The exact gnmx branch that skips the re-emit is inferred from behavior, not
disassembled — a future watchpoint on the fence reads inside native gnmx would pin the precise instruction.
Both ACs are met (it was NOT a provenance bug; the atlas now reaches the shader). DO NOT MERGE per instructions.
